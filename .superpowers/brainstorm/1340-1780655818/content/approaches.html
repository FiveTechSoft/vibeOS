3 Enfoques de Arquitectura para vibeOS

Stack confirmado: Harbour + DeepSeek (vía componentes CCHarbour). Escritorio completo multi-ventana.

Componentes Disponibles (ya existen, no hay que reescribir)

🧠 ccapi.prg
Cliente DeepSeek streaming SSE. Chat completions con tool calls.
🌐 cchttp.prg
HTTP via curl. Soporta POST streaming y GET/POST acumulado.
📡 hbhttpd
Servidor HTTP multihilo. Sesiones, routing, keep-alive, SSL.
🔧 ccconfig.prg
Resolución de API key, base URL, proveedor.
A

Monolito Harbour (RECOMENDADO)

Un solo .exe con servidor HTTP embebido (hbhttpd) + cliente DeepSeek (ccapi). Sirve estáticos (CSS XP, JS) y API en mismo proceso. Sesiones hbhttpd mapean 1:1 a windowSessions del LLM. ~3-4MB binario final.

Pros

  • Un solo binario, un solo proceso
  • Sesiones nativas = estado de ventanas
  • Mínimo overhead, máxima velocidad
  • Fácil deploy (copiar .exe + assets/)

Cons

  • Todo en Harbour (curva si no conoces)
  • Sin hot-reload de frontend
B

Harbour API + Frontend Separado

Harbour backend solo como API (DeepSeek proxy + estado). Frontend servido por separado (Node/Vite o nginx). Comunicación vía WebSocket o SSE.

Pros

  • Separación clara front/back
  • Hot-reload en desarrollo
  • Frontend puede usar cualquier tooling JS

Cons

  • Dos procesos que gestionar
  • Necesita WebSocket/SSE extra
  • Más complejidad de deploy
C

Mínimo: Harbour sirve un solo HTML generado

Harbour genera HTML inicial completo con sistema LLM. El frontend carga ese HTML y hace fetch al backend para deltas. Sin assets estáticos separados.

Pros

  • Extremadamente simple
  • Todo inline, sin archivos CSS/JS separados

Cons

  • CSS y JS embebidos en cada request
  • No aprovecha caché del navegador
  • Difícil debuggear frontend
